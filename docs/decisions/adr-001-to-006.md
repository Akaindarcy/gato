# Architecture Decision Records (ADRs)

## ADR-001: Offline-First with Drift + Custom Sync Engine

**Status:** Accepted  
**Context:** GATO must work in congested markets where internet is unreliable. Users need to record sales even when offline, then sync later.  
**Decision:** Use Drift (SQLite) for local storage with a custom sync engine over Supabase Realtime + REST. Each entity write posts to a local `sync_queue` table. A background sync worker retries with exponential backoff and conflict resolution rules (server-wins with local log).  
**Consequences:** 
- Pro: Full offline functionality; no reliance on always-on connectivity.  
- Con: Increased complexity in conflict handling and schema migration across local + remote.

## ADR-002: Riverpod for State Management

**Status:** Accepted  
**Context:** Need compile-safe, testable state management that doesn't depend on BuildContext.  
**Decision:** Use Riverpod 2.x with StateNotifier providers per feature.  
**Consequences:** 
- Pro: Great testability, no widget coupling, widespread adoption.  
- Con: Learning curve for newer Flutter devs.

## ADR-003: Phone OTP via Supabase Auth

**Status:** Accepted  
**Context:** Target users have no email; email-based auth would block adoption.  
**Decision:** Supabase phone OTP for signup/login. Store phone as primary identifier.  
**Consequences:** 
- Pro: Low friction; SMS delivered via Africa's Talking or telco.  
- Con: Costs per OTP; handling phone number changes requires re-verification.

## ADR-004: Rule-Based AI Coach for MVP

**Status:** Accepted  
**Context:** External LLM APIs cost money and require connectivity.  
**Decision:** MVP uses rule-based engine (AICoachService) that computes insights from local transaction data. Future migration to on-device small model (e.g., TensorFlow Lite) or edge LLM.  
**Consequences:** 
- Pro: Free, fast, offline-capable.  
- Con: Limited personalization; not truly conversational.

## ADR-005: Voice Input Using Flutter TTS + STT

**Status:** Accepted  
**Context:** Users have low literacy; voice is critical.  
**Decision:** Use `flutter_tts` and `speech_to_text` plugins. Kirundi support via `fr-RW` locale as fallback; Kinyarwanda via `rw-RW`.  
**Consequences:** 
- Pro: Works offline; no API dependency.  
- Con: STT accuracy depends on device and locale; may need custom model tuning later.

## ADR-006: Single Supabase Project with RLS

**Status:** Accepted  
**Context:** Need multi-tenant data isolation (business_id).  
**Decision:** All tables in one Supabase Postgres database with Row Level Security (RLS) policies keyed on `business_id`. Edge functions for nightly Trust Score computation and partner integrations.  
**Consequences:** 
- Pro: Simplified operations; single connection string.  
- Con: Scaling beyond ~1M rows may require sharding later (not in 3-year horizon).
